# ExecPlan: ガチャ電話（プランB：Whisper + Gemini + TTS 構成）へのリファクタリング

## 目的
アプリを「ガチャ電話」としてリニューアルし、不満点であった「音声認識の精度（英語が聞き取られない）」および「レスポンスの遅さ」を解消する。
プランB（Whisper STTによる高精度認識）に基づき、アーキテクチャを刷新する。

## 目標 (Goals)
1. **高精度音声認識 (STT) の導入**: Groq API (Whisper-large-v3) 等を用いて、日本人が話す英語でも正確かつ高速に認識できるようにする。
2. **UI/UX の改善**: 音声入力のタイミングをユーザーがコントロールできるUI（例：Hold to Talk / トランシーバー形式）または、より短い無音判定を導入し、レスポンスの遅延感をなくす。また「Thinking...」などのフィードバックを表示する。
3. **カストーマスの撲滅（プロンプトの強力な固定化）**: キャラクター設定（System Prompt）を強制し、会話履歴を正しく管理することで、キャラクターのブレをなくす。
4. **TTS（音声合成）の実装**: AIのテキスト返答を自然な音声で読み上げるTTSを統合する。

## 前提条件 (Prerequisites)
- [ ] Groq APIキーの取得（STT用。Whisperが高速かつ低コストで利用可能）
- [ ] （オプション）最適なTTSサービス（OpenAI TTSやElevenLabs、またはWeb Speech APIなどコストと品質のバランス）の選定とAPIキーの取得。※今回はコスト重視のため、まずは安価なOpenAI TTSか、ブラウザ標準のWeb Speech APIをフォールバックとして検討。
- [ ] 環境変数への設定（`.env.local`）

## 実装ステップ (Steps)

### Step 1: 必要なAPIキーの準備とパッケージ追加
- ユーザーに Groq APIキー および TTS用のAPIキー（OpenAI等）を準備してもらう。
- 必要なnpmパッケージ（`groq-sdk`, `openai` 等）をインストールする。
- 環境変数を更新する。

### Step 2: 音声録音・送信UIの改修（トランシーバー形式）
- `App.tsx` または音声録音コンポーネントを改修。
- GeminiのLive API（Websocket）通信から、単発のREST API通信（録音 -> STT -> LLM -> TTS -> 再生）のフローに変更する。
- 「押している間だけ録音する（Hold to Talk）」または「タップで録音開始・終了」するUIに変更し、無音判定による不要な待ち時間を解消する。

### Step 3: STT (Whisper) と LLM (Gemini) の連携実装
- `src/services/gemini.ts` を改修。
- ユーザーの録音データをGroq Whisper APIに送信してテキスト化する関数を実装。
- テキスト化されたユーザーの入力を、強力に固定化されたSystem Promptと会話履歴とともにGemini 1.5 Flash（またはGroq LLM）に送信し、テキストの返答を得る。

### Step 4: TTS (音声合成) の連携と再生実装
- LLMから得られたテキスト返答をTTS API（OpenAI等）に送信し、音声データ（mp3等）を取得する処理を実装。
- フロントエンドで音声データを再生すると同時に、テキストを画面に表示する。
- ローディング中（Thinking...）の表示など、UXを向上させる。

### Step 5: 全体テストと微調整
- 英語・日本語での認識精度をテストし、聞き取りのストレスが解消されたか確認。
- キャラクターのブレ（カストーマス）が発生しないか、プロンプトと履歴の挙動をテスト。
- 生成AIの「ガチャ要素」を楽しめるよう、プロンプトの調整を行う。
